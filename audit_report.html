PART 3 Financial Data Integrity Audit
Verification of all formulas, calculations, and data sourcing
BUG-02 · CRITICAL · FINANCIAL FORMULA ERROR
NOI Double-Deducts Management Fee
Current formula: noi = gop - managementFee - (revenue × 0.015)
Problem: managementFee in data.js is already computed as 10% of revenue. The formula then additionally deducts 1.5% of revenue as an "operator incentive" on top. This means NOI understates by ~Rp 12.7M per month (for Feb 2026). For a CFO document being shown to Owner, this is a material financial misstatement.
Correct formula: noi = gop - managementFee (managementFee already captures the full non-operating deduction)
Fixconst noi = d.gop.actual - d.managementFee; // Remove the -(d.revenue.actual*0.015)
BUG-07 · HIGH · METRIC CALCULATION ERROR
Cash Runway Massively Overstated
Current formula: runway = cashPosition / (labourCost / 30)
Problem: Labour cost is ~Rp 253M/month but total operating costs are ~Rp 660M/month (revenue − GOP). Using only labour cost as the burn rate produces a runway of 661 days (Feb 2026) when the correct runway — against total operating costs — is approximately 253 days. The 661-day figure, if shown to an Owner, would dangerously misrepresent liquidity risk.
Correct formula: Use total cost = revenue − GOP as the monthly burn denominator.
Fixconst totalMonthlyCost = d.revenue.actual - d.gop.actual;
const runway = Math.floor(d.cashPosition / (totalMonthlyCost / 30));
BUG-VAR-01 · MEDIUM · DATA SOURCING ERROR
Variance Report Budgets Are Circular (Derived from Actual)
In both the P&L and Variance Analysis reports, Rooms Revenue budget = roomsRevenue × 1.10 and F&B budget = fbRevenue × 1.08. These are not sourced from data.js — they are algebraically derived from the actual figures.
This creates a mathematically circular comparison that always shows a fixed percentage variance, regardless of true performance. A CFO document comparing "Actual vs Budget" must use real budget figures.
FixAdd roomsRevenueBudget and fbRevenueBudget fields to each monthly record in data.js, then source from d.roomsRevenueBudget and d.fbRevenueBudget.
AUDIT-GOPPAR · MEDIUM · DATA ACCURACY
GOPPAR Value Inconsistency (Jan 2026)
data.js records Jan 2026 GOPPAR = 43,000. Calculated: GOP (195,000,000) ÷ (150 rooms × 31 days) = 41,935. Discrepancy: Rp 1,065. Minor but worth correcting for data integrity — GOPPAR is a key benchmarking metric.

BUG-03 · CRITICAL · DOM STATE CORRUPTION

exportPDF() calls s.style.display='block' on all 5 meeting sections to force them into the print window's HTML. It never restores the original display state. After export, all 5 sections show simultaneously instead of one at a time. goSec(n) uses classList.toggle('active'...) but the inline style.display overrides the CSS class — making section navigation permanently broken until page reload.
FixUse cloneNode(true) to copy sections, not set display on originals:

PART 7 Code Stability Review
Assessment of JavaScript structure, fragility, scalability, and chart management
STABILITY-01 · MEDIUM
Chart Re-render Risk — No Per-Canvas Destroy
renderExplore() calls Object.keys(charts).forEach(k=>dChart(k)) — this destroys ALL tracked charts at once before re-render. This works but is fragile: if a chart registration mismatches its canvas ID (e.g., in Meeting Mode), it won't be destroyed, causing "Canvas already in use" Chart.js errors on re-navigation. The meeting charts (m-rev, m-cash) are destroyed separately in rMtg2Charts() which is correct but creates two separate chart lifecycle systems.
STABILITY-02 · MEDIUM
Period Mode Filter (YTD / Full Year / T12M) Not Implemented
The period mode selector offers 4 options (Monthly, YTD, Full Year, Trailing 12M) but S.pMode is never read by any renderer. Users who select "YTD" will see unchanged monthly data — a confusing dead state. Either implement the aggregation logic or remove the options until implemented.
STABILITY-03 · INFO
Scalability: New Months — No Work Required
Adding a new month to PORTAL_DATA.monthly in data.js and adding its key to the period dropdown is all that's needed. Charts automatically include it via recent(). Good design — the system scales well for monthly updates.
STABILITY-04 · INFO
No Error Boundaries — Any JS Error Kills Entire App
There are no try/catch blocks around chart renders, data accessors, or meeting section renders. A single malformed data.js entry (missing field, wrong type) will throw an uncaught error and freeze the entire portal. A CFO in a live meeting scenario would see a broken blank screen.
